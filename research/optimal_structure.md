## 📝 Naming Convention: `snake_case` + `_intent.yaml`

| Level | Pattern | Example | Location |
|-------|---------|---------|----------|
| **System** | `system_intent.yaml` | `system_intent.yaml` | Project root |
| | `{project}_system_intent.yaml` | `book_system_intent.yaml` | Project root |
| **Workflow** | `{workflow}_intent.yaml` | `lead_qualification_intent.yaml` | `workflows/` (recommended) or `{coordinator_dir}/` (alternative) |
| **Module** | `{module}_intent.yaml` | `lead_qualifier_intent.yaml` | `{module}/` (required for tooling) |
| **Task** | `{function}_intent.yaml` | `qualify_lead_intent.yaml` | `{module}/intents/` (required for tooling) |
| **Variant** | `{variant}_{module}_intent.yaml` | `erik_reviewer_intent.yaml` | `{module}/` |

**Workflow placement:** **Recommended:** `workflows/{workflow_name}_intent.yaml` (discoverability). **Alternative:** `{coordinator_dir}/{workflow_name}_intent.yaml` when the workflow is tightly coupled to a single orchestrator and not referenced independently. See `framework.md` "Workflow-Level Intents" for when to choose which.

## ✅ DO (Research-Backed)

### Placement
- ✅ **Co-locate intent with implementation** (Next.js, Nx, Microsoft)
- ✅ **System intents at project root** (conventional)
- ✅ **Workflow intents:** **Recommended** `workflows/` (discoverability: `find workflows/ -name "*_intent.yaml"`). **Alternative:** alongside coordinator file when workflow is tightly coupled to a single orchestrator and will not be referenced independently (co-location with coordinator).
- ✅ **Module intents in module folder** (proximity)
- ✅ **Task intents in `intents/` subfolder** (organization)

### Naming
- ✅ **Use lowercase snake_case** (YAML community standard)
- ✅ **Use `_intent.yaml` suffix** (discoverability and tool compatibility)
- ✅ **Module naming:** `{module}_intent.yaml` (not bare `intent.yaml` - tools require the suffix for discovery)
- ✅ **Task placement:** `{module}/intents/{task}_intent.yaml` (tools detect task-level by `intents/` in path)
- ✅ **Descriptive names** (`lead_qualification` > `lq`)
- ✅ **Consistent across project** (all lowercase)

### Content
- ✅ **Structured YAML** (AI agent parsing)
- ✅ **Semantic keys** (`intent`, `constraints`, `success_criteria`)
- ✅ **Executable tests** (verification)
- ✅ **Optional metadata for large projects** (`feature_id`, `category`, `tags`, `status`) — enables derived feature inventory and duplicate prevention; inventory is generated by tooling, not a separate file.

## ❌ DON'T (Anti-Patterns)

### Placement
- ❌ **Centralized `/intents` folder at root** (one folder for all intent levels—violates co-location). Note: `workflows/` is a level-specific folder for workflow-level intents only, not this anti-pattern.
- ❌ **Separate `/docs` folder** (drift risk)
- ❌ **Wiki or external docs** (not version-controlled with code)
- ❌ **Flat task intents** (place in `{module}/intents/` for tool detection)

### Naming
- ❌ **UPPERCASE** (`MASTER_INTENT.yaml`) - non-standard
- ❌ **camelCase** (`masterIntent.yaml`) - not YAML standard
- ❌ **kebab-case for keys** (snake_case is YAML standard)
- ❌ **Generic names** (`doc.yaml`, `config.yaml`)
- ❌ **No suffix** (`intent` alone - conflicts with directories)
- ❌ **Bare `intent.yaml` for modules** (use `{module}_intent.yaml` for tool discovery)

## 🤖 AI Agent Discovery (2026 Research)

### How AI Agents Find Intent Artifacts

**Primary methods (ranked by effectiveness):**

1. **Semantic Search (Most Effective)**
   - Agents use embeddings and RAG to search by meaning
   - Example: Search for "user requirements" finds intent files
   - **Key:** Semantic content matters more than filename

2. **Workspace Indexing (Very Effective)**
   - Co-located files indexed together as clusters
   - When agent searches "CSV export context," finds code + intent + tests
   - **Key:** Co-location is critical

3. **Symbol Indexing (Very Effective)**
   - YAML structure parsed and indexed
   - Agents can search for specific keys (`constraints`, `success_criteria`)
   - **Key:** Structured YAML format matters

4. **Dynamic Retrieval (Effective)**
   - Agents pull context as needed during conversation
   - Not limited to initial context window
   - **Key:** Intent artifacts accessible throughout development

5. **Filename Patterns (Minor Factor)**
   - Consistent naming helps (agents learn the pattern)
   - Uppercase vs lowercase: minimal impact
   - **Key:** Consistency matters, specific case less so

## 🎯 Optimal Workflow for AI Agents

```
USER REQUEST:
  "Add CSV export for admins"

1. AGENT DISCOVERS CONTEXT (Semantic Search + Proximity)
   → Finds: agent/csv_exporter/csv_exporter_intent.yaml (co-located with code)
   → Indexes: constraints, success_criteria, tests

2. AGENT READS INTENT (YAML Parsing)
   → Extracts: admin_only=true, columns=[...], format=ISO 8601
   
3. AGENT ASKS CLARIFYING QUESTIONS (Intent-Driven Dialogue)
   → "Intent specifies ISO 8601. UTC or local timezone?"
   → "Intent says <2s for 10k rows. Should I add pagination?"

4. AGENT IMPLEMENTS (Constraint-Driven)
   → Writes code that satisfies constraints
   → Creates tests from success_criteria

5. AGENT VERIFIES (Automated Testing)
   → Runs constraint tests
   → Reports: "Implementation satisfies 4/4 constraints"

6. AGENT EXPLAINS (Intent Reference)
   → "I implemented CSV export per agent/csv_exporter/csv_exporter_intent.yaml"
   → "All constraints verified: admin-only, correct columns, ISO dates, <2s"
```

## 🔄 Adopting IVD in Existing Projects (Brownfield)

### Bootstrap with ivd_init

For projects with existing code and documentation:

```bash
# 1. Initialize IVD - creates system intent with project context
ivd init --project-root /path/to/your/project

# This auto-scans and captures:
# - Code rules (.cursorrules, lint configs)
# - Architecture (ARCHITECTURE.md, ADRs, principles)
# - Tools/scripts (key scripts, CLIs)
# - Libraries to reuse (internal shared libs)
# - Key paths ("map to the stars": entrypoints, modules, workflows, docs, tests)
# - Existing docs (README, API.md, CONTRIBUTING)

# 2. Review and enrich system_intent.yaml
# Add architecture principles not auto-detected
# Add conventions, libraries_reuse with when_to_use guidance

# 3. Create intents for critical modules (start with 3-5)
ivd scaffold --level module --name lead_scorer --module-path agent/lead_scorer
# Auto-links: parent_intent: "../../system_intent.yaml"

# 4. Expand gradually as you modify modules
# Don't create intents for entire codebase at once
```

### Project Context and "Map to the Stars"

System intents in existing projects should include `project_context`:

```yaml
# system_intent.yaml

project_context:
  code_rules:
    - path: ".cursorrules"
      description: "Cursor IDE rules"
  
  architecture:
    - path: "docs/ARCHITECTURE.md"
      description: "System architecture"
    - principle: "Event-driven with async messaging"
  
  tools_scripts:
    - path: "scripts/deploy.sh"
      description: "Deployment automation"
  
  libraries_reuse:
    - library: "internal/logging"
      when_to_use: "All modules (use log_context decorator)"
  
  key_paths:  # "Map to the stars"
    entrypoints:
      - path: "api/main.py"
        description: "FastAPI application"
    modules:
      - path: "agent/"
        description: "AI agents"
    workflows:
      - path: "workflows/"
        description: "Business processes"
    docs:
      - path: "docs/"
        description: "Documentation"
    tests:
      - path: "tests/"
        description: "Test suites"
  
  existing_docs:
    - path: "README.md"
      description: "Project overview"
```

**Child intents reference parent** via `parent_intent` to inherit project context.

## 🔄 Migration Path (Renaming Old Intents)

If you have existing projects with old naming:

```bash
# 1. Rename system-level intents
mv MASTER_INTENT.yaml system_intent.yaml
mv PROJECT_NAME_INTENT.yaml {project}_system_intent.yaml

# 2. Update all references
find . -name "*.md" -o -name "*.yaml" | xargs sed -i '' 's/MASTER_INTENT\.yaml/system_intent.yaml/g'
find . -name "*.md" -o -name "*.yaml" | xargs sed -i '' 's/IVD_FRAMEWORK_INTENT\.yaml/ivd_system_intent.yaml/g'
find . -name "*.md" -o -name "*.yaml" | xargs sed -i '' 's/IVD_BOOK_INTENT\.yaml/book_system_intent.yaml/g'

# 3. Verify no old references remain
grep -r "MASTER_INTENT\.yaml\|IVD_.*_INTENT\.yaml" . --include="*.md" --include="*.yaml" | grep -v "research"
# Should return: (empty)

# 4. Update version control
git mv MASTER_INTENT.yaml system_intent.yaml
git add .
git commit -m "Align with research-backed naming standards (2026)"
```
